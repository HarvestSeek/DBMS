    2. function dependency

    1. 函数依赖

    e.g.2 $\small\mathit{instructor}(\mathit{\textcolor{black}{ID},name,dept\_name,salary})$ $\small\mathit{department}(\mathit{\textcolor{black}{dept\_name},building,budget})$
    假设用它们的自然连接 $\small\mathit{inst\_dept}\ (\mathit{\textcolor{black}{ID},name,salary,dept\_name,building,budget})$ 取代上面的两个模式，会导致以下问题：

    • 数据冗余：一个系对应多名教师，$\small\mathrm{department}$ 的信息被冗余存储
    • 插入和删除：无法插入没有教师信息的系，删除教师也会连带着删除系的信息
    • 更新：更新 $\small\mathrm{department}$ 的信息时，可能需要更新大量元组
    尽管在 E-R 设计过程中避免了 $\small\mathit{inst\_dept}$ 的产生，但这只是一种幸运的情况。因此，需要一种方法判断 $\small\mathit{inst\_dept}$ 不是一个好的设计:
     假如有模式 $\small(\mathit{dept\_name,budget})$，则 $\small\mathit{dept\_name}$ 可作为主码 ($\small\mathit{dept\_name\to budget}$)。且 $\small\mathit{dept\_name}$ 不是 $\small\mathit{inst\_dept}$ 的主码，才产生了上面的问题。

    在下面的论述中，用希腊字母表示属性集，用 $\small r(R)$ 表示关系模式。

    (1). 定义

    考虑关系模式 $\small r(R)$，其中 $\small\alpha\subseteq R$，$\small\beta\subseteq R$。

    • 给定 $\small r(R)$ 的一个实例 $\small r'$，若对实例 $\small r'$ 中任意的一对元组 $\small t_1$ 和 $\small t_2$，满足
      $\small\mathbf{if}\ \ t_1[\alpha]=t_2[\alpha]\ \ \mathbf{then}\ \ t_1[\beta]=t_2[\beta]$，则实例 $\small r'$ 满足函数依赖 $\small\alpha\to\beta$
    • 如果 $\small r(R)$ 的每个合法实例都满足函数依赖 $\small\alpha\to\beta$，就说函数依赖在 $\small r(R)$ 上成立 (hold)
    • 若 $\small K\to R$ 在 $\small r(R)$ 上成立，则 $\small K$ 是 $\small r(R)$ 的一个超码
    • 若函数依赖 $\small\alpha\to\beta$ 满足 $\small\beta\subseteq\alpha$，称此函数依赖是平凡的 (trivial)

    需要注意，某些函数依赖可能在一个关系模式的实例上成立，但它不一定在关系模式上成立，判别时需要考虑关系模式被赋予的意义，考虑所有可能的实例。

    • 若 $\small r(R)$ 的每一个满足函数依赖集 $\small F$ 的实例也满足函数依赖 $\small f$，称 $\small F$ 逻辑蕴含 (imply) $\small f$
    • $\small r(R)$ 上 $\small F$ 逻辑蕴含的所有函数依赖组成的集合为 $\small F$ 的闭包，记为 $\small F^+$
    • $\small\alpha\to\beta$ 逻辑蕴含 $\small\alpha\cup\lbrace\dots\rbrace\to\beta-\lbrace\dots\rbrace$
    部分/完全/传递函数依赖：
    • 设有函数依赖 $\small\alpha\to\beta$，若对 $\small\forall\gamma\subset\alpha$，都有 $\small\gamma\to\beta$ 不成立，就称 $\small\beta$ 完全函数依赖于 $\small\alpha$，否则称为部分函数依赖。
    • 若有 $\small\alpha\to\beta$ 和 $\small\beta\to\gamma$，就称 $\small\gamma$ 传递函数依赖于 $\small\alpha$。

    (2). 运算律

    Armstrong 公理：

    • 自反律 (reflexivity)：若 $\small\beta\subseteq\alpha$，则 $\small\alpha\to\beta$ 成立
    • 增补律 (augmentation)：若 $\small\alpha\to\beta$ 成立，则 $\small\gamma\alpha\to\gamma\beta$ 成立
    • 传递律 (transitivity)：若 $\small\alpha\to\beta$ 和 $\small\beta\to\gamma$ 成立，则 $\small\alpha\to\gamma$ 成立
    上述公理是完备的，它们能产生全部 $\small F^+$，但会很麻烦。
    • 合并律 (union)：若 $\small\alpha\to\beta$ 和 $\small\alpha\to\gamma$ 成立，则 $\small\alpha\to\beta\gamma$ 成立
    • 分解律 (decomposition)：若 $\small\alpha\to\beta\gamma$ 成立，则 $\small\alpha\to\beta$ 和 $\small\alpha\to\gamma$ 成立
    • 伪传递律：若 $\small\alpha\to\beta$ 和 $\small\gamma\beta\to\delta$ 成立，则 $\small\alpha\gamma\to\delta$ 成立

    (3). 属性集的闭包

    若 $\small\alpha\to B$，称属性 $\small B$ 被 $\small\alpha$ 函数确定 (functionally determine)。
    函数依赖集 $\small F$ 下所有被 $\small\alpha$ 函数确定的属性集合称为 $\small F$ 下 $\small\alpha$ 的闭包，记为 $\small\alpha^+$。 

    result = α
do
    for each β → γ in F
        if β ⊆ result
            result = result ∪ γ
while(result 不变)
    右侧算法的时间为 $\small O(|F|^2)$。存在一个 $\small O(|F|)$ 的算法，暂不介绍。
    属性闭包算法可用于：
    • 为判断 $\small\alpha$ 是否为超码，检查是否 $\small R\subseteq\alpha^+$
    • 为判断 $\small\alpha\to\beta$ 是否成立，检查是否 $\small \beta\subseteq\alpha^+$
    • 计算 $\small F^+$: 对任意 $\small\gamma\subseteq R$，求 $\small\gamma^+$；对任意 $\small S\subseteq\gamma^+$，输出一个函数依赖 $\small\gamma\to S$

    (4). 正则覆盖

     在关系上执行更新时，数据库必须保证所有函数依赖都不被破坏。为减小冲突检测的开销，我们需要求函数依赖集的简化集 $\small F'$，$\small F'$ 和 $\small F$ 具有相同的闭包，数据库满足 $\small F'$ 等价于满足 $\small F$。

    考虑函数依赖集 $\small F$ 和函数依赖 $\small\alpha\to\beta$，无关属性 (extraneous) 的形式化定义如下：

    • (1). 若 $\small a\in\alpha$ 且 $\small F$ 逻辑蕴含 $\small(F-\lbrace\alpha\to\beta\rbrace)\cup(\lbrace(\alpha-a)\to\beta\rbrace)$，则属性 $\small a$ 在 $\small\alpha$ 中是无关的
    • (2). 若 $\small a\in\beta$ 且 $\small(F-\lbrace\alpha\to\beta\rbrace)\cup(\lbrace\alpha\to(\beta-a)\rbrace)$ 逻辑蕴含 $\small F$ ，则属性 $\small a$ 在 $\small\beta$ 中是无关的

    上述逻辑蕴含的方向反过来都是自然成立的, 因此若满足上述要求，有 $\small F\Leftrightarrow F'$，则数据库满足 $\small F\Leftrightarrow$ 数据库满足 $\small F'$
    检验无关属性的计算方法是:

    • (1). 判断在 $\small F$ 中是否有 $\small\beta\subseteq(\alpha-a)^+$
    • (2). 判断在 $\small F'$ 中是否有 $\small a\in\alpha^+$
    e.g. 若 $\small F$ 中有 $\small AB\to C$ 和 $\small A\to C$，那么 $\small B$ 在 $\small AB\to C$ 中是无关的
    e.g. 若 $\small F$ 中有 $\small AB\to CD$ 和 $\small A\to C$，那么 $\small C$ 在 $\small AB\to CD$ 中是无关的
    e.g. 若 $\small F$ 中有 $\small A\to B$、$\small B\to C$ 和 $\small A\to C$，那么 $\small C$ 在 $\small A\to C$ 中是无关的，需删除 $\small A\to C$

    $\small F$ 的正则覆盖 (canonical cover) 记为 $\small F^c$，是一个满足下列性质的依赖集：

    • $\small F$ 逻辑蕴含 $\small F^c$ 中所有依赖，$\small F^c$ 逻辑蕴含 $\small F$ 中所有依赖
    • $\small F^c$ 中所有函数依赖不含无关属性
    • $\small F^c$ 中不存在两个函数依赖 $\small\alpha_1\to\beta_1$ 和 $\small\alpha_2\to\beta_2$，满足 $\small\alpha_1=\alpha_2$
    $\small F^c$ 的计算方法是，在循环中不断合并形如 $\small\alpha_1\to\beta_1$ 和 $\small\alpha_1\to\beta_2$ 的项，并检测每个项是否有多余属性，直到收敛。

    (5). 最小覆盖

     最小覆盖 (minimal cover) 同样是不断消除函数依赖中的无用属性直到收敛，和正则覆盖不同的是，极小覆盖把所有形如 $\small\alpha\to a_1a_2\dots a_n$ 的项分解为 $\small\alpha\to a_1,\ ..., \alpha\to a_n$，因此每个极小函数依赖项右侧只有一个属性。极小函数依赖集是不唯一的。

    e.g. 若 $\small F=\lbrace A\to B,\ B\to A,\ B\to C,\ A\to C,\ C\to A\rbrace$，则 $\small F$ 的最小函数依赖集可以为
        $\small F_1=\lbrace A\to B,\ B\to C,\ C\to A\rbrace$ 或 $\small F_2=\lbrace A\to B,\ B\to A,\ A\to C,\ C\to A\rbrace$

    (6). 无损分解

    假设有关系模式 $\small\mathit{employee}\ (\mathit{ID,name,street,city,salary})$，如果把它分解为 $\small\mathit{employee1}\ (\mathit{ID,name})$ 和 $\small\mathit{employee2}\ (\mathit{name,street,city,salary})$，由于可能存在重名的雇员，$\small\mathit{employee1}⋈\mathit{employee2}$ 的元组会比原本的 $\small\mathit{employee}$ 更多。这样的分解称为有损分解。

    如果用两个关系模式 $\small r(R_1)$ 和 $\small r(R_2)$ 代替 $\small r(R)$ 时没有信息损失，即 $\small\Pi_{R_1}(r)⋈\Pi_{R_2}(r)=r$，则称此分解为无损分解 (lossless decomposition)。
     把 $\small R$ 分解为 $\small R_1$ 和 $\small R_2$ 时，如果函数依赖 $\small R_1\cap R_2\to R_1$ 和 $\small R_1\cap R_2\to R_2$ 中至少有一个在 $\small F^+$ 中，即 $\small R_1\cap R_2$ 是 $\small R_1$ 或 $\small R_2$ 的超码，那么此分解是无损的。

    (7). 保持依赖

    令 $\small F$ 为模式 $\small R$ 上的一个函数依赖集，$\small R_1,\ \dots\ ,R_n$ 为 $\small R$ 的一个分解。$\small F$ 在 $\small R_i$ 上的限定是 $\small F^+$ 上所有只包含 $\small R_i$ 中的属性的集合 $\small F_i$。
    令 $\small F'=F_1\cup F_2\cup\ \dots\ \cup F_n$，如果 $\small F'^+=F^+$，就称此分解为保持依赖的分解 (dependency-preserving).

    注意定义限定时使用的是 $\small F^+$ 中的所有依赖而不仅仅是 $\small F$。假设 $\small F=\lbrace A\to B,\ B\to C\rbrace$ 有一个到 $\small\mathit{AC}$ 和 $\small\mathit{AB}$ 的分解，$\small F$ 在 $\small\mathit{AC}$ 上的限定包含 $\small A\to C$。
    已知 $\small F$ 逻辑蕴含 $\small F'$，要判别 $\small F'^+=F^+$，就等价于在 $\small F'$ 中判断 $\small F$ 的每个 $\small\alpha\to\beta$ 是否被保持，即判断 $\small\beta\subseteq\alpha_{F'}^+$ 是否成立。

    2. 范式

    (1). 原子域和第一范式

    如果一个域的元素都是不可分的，那么这个域是原子的；如果关系模式 $\small R$ 的所有属性的域都是原子的，那么 $\small R$ 属于第一范式 (First Normal Form, 1NF)

    e.g. 某机构给雇员分配形如 $\small\mathrm{CS012}$ 和 $\small\mathrm{EE127}$ 的雇员号，其中包含了机构和编号两部分信息。从数据库的角度看，如果应用程序没有把标识号拆开并解析系名 (例如单独另设一个属性)，就可以认为这个域是原子的。
    在关系数据库中，1NF 是最起码的要求，下面的讨论都在 1NF 的范围内。

    (2). 第二范式

    若关系模式 $\small r(R)$ 是 1NF，且每个非主属性都完全函数依赖于 $\small r(R)$ 的候选码，称 $\small r(R)$ 属于 2NF。

    e.g.1 设有关系模式 $\small\mathit{stu\_course}\ (\mathit{stu\_id,course\_id,stu\_dept,stu\_loc,grade})$，其依赖包括
    $\small(\mathit{stu\_id,course\_id})\to\mathit{grade},\ \ \ \ \ \ \ \mathit{stu\_id\to stu\_dept}$
    $\small(\mathit{sdu\_id,course\_id})\to\mathit{stu\_dept},\ \mathit{stu\_id\to stu\_loc}$
    $\small(\mathit{stu\_id,course\_id})\to\mathit{stu\_loc},\ \ \ \ \mathit{stu\_dept\to stu\_loc}$ (一个系的人住在一个地方)
    非键属性 $\small\mathit{stu\_loc、stu\_dept}$ 对候选码部分函数依赖，因此将其分解为
     $\small(\mathit{stu\_id,course\_id,grade})$  和  $\small(\mathit{stu\_id,stu\_dept,stu\_loc})$

    (3). 第三范式

    若 $\small r(R)$ 是 2NF，且它的任何一个非主属性都不传递依赖于任何候选码，称 $\small r(R)$ 属于 3NF。

    续 e.g.1 对分解后得到的 $\small(\mathit{stu\_id,stu\_dept,stu\_loc})$，每个非键属性都完全函数依赖于候选码，但存在一个到候选码的传递函数依赖 $\small\mathit{stu\_id\to stu\_dept\to stu\_loc}$，因此将其分解为
     $\small(\mathit{stu\_id,stu\_dept})$  和  $\small(\mathit{stu\_dept,stu\_loc})$。

    3NF 的条件也可按如下方式定义。对 $\small F^+$ 中所有形如 $\small\alpha\to\beta$ 的函数依赖，以下至少一项成立：

    • $\small\alpha\to\beta$ is trivial
    • $\small\alpha$ 是 $\small R$ 的一个超码
    • $\small\beta-\alpha$ 的每个属性 $\small a$ 都包含于 $\small R$ 的一个候选码中 (不同属性可能包含于不同的候选码)

    (4). Boyce-Codd 范式

    BCNF 具有以下性质：

    • 所有非键属性都完全依赖于某个候选码
    • 所有键属性都完全依赖于某个不包含它的候选码
    • 没有任何属性函数依赖于非键属性

    e.g.2 设有关系模式 $\small\mathit{sic}\ (\mathit{stu\_id,inst\_id,cousre\_id})$，并假设每个教师只教一门课，每门课由若干教师教，每名学生选定一门课后就确定一个固定的教师。由此可确定函数依赖：
    $\small(\mathit{stu\_id,course\_id})\to\mathit{inst\_id}$， $\small(\mathit{stu\_id,inst\_id})\to\mathit{course\_id}$， $\small\mathit{inst\_id}\to\mathit{course\_id}$
    因此，$\small(\mathit{stu\_id,course\_id})$ 和 $\small(\mathit{stu\_id,inst\_id})$ 都可作候选码，$\small\mathit{sic}$ 属于 3NF。尽管如此，由于 $\small\mathit{inst\_id}\to\mathit{course\_id}$ 的存在，此关系模式仍存在插入删除、数据冗余等问题，继续将其分解为
     $\small(\mathit{stu\_id,course\_id})$  和  $\small(\mathit{inst\_id,course\_id})$。

    BCNF 在 3NF 的基础上进行限制。对 $\small F^+$ 中所有形如 $\small\alpha\to\beta$ 的函数依赖，以下至少一项成立：

    • $\small\alpha\to\beta$ is trivial
    • $\small\alpha$ 是 $\small R$ 的一个超码
    如果一个关系数据库中的所有关系模式都属于 BCNF，那么它在函数依赖范畴内实现了模式的彻底分解，达到了最高规范化程度，消除了插入和删除异常。

    3. 分解算法

    (1). 3NF 分解

     Algorithm1: 保持依赖的 3NF 分解算法                                   
Input: r(R), F, 其中 F 是最小覆盖
Output: R 的一个保持依赖分解 ρ, ρ 的每
    个关系模式都是     F 在该关系上投影的 3NF
1. 把 R 中不出现在 F 中的属性去掉，并组成单一模式
2. 对 ∀X → A ∈ F, 以 XA 组成模式; 若有 X → A1, ... , 
   X → Am 都属于 F, 则以 XA1...Am 组成模式
3. 取 ρ 为上述模式的集合

    上述算法只保证相应的 3NF 分解是保持依赖的。如果分解 $\small\rho$ 中某个集合包含了 $\small R$ 的一个候选键，$\small\rho$ 也是无损连接的。因此，若需要无损分解且 $\small\rho$ 中没有候选键，在算法 1 的基础上追加一步 $\small\rho=\rho\cup\lbrace X\rbrace$，$\small X$ 是 $\small R$ 的某一个候选键。
