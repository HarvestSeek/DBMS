    2. function dependency

    1. 原子域和第一范式

    如果一个域的元素都是不可分的，那么这个域是原子的；如果关系模式 $\small R$ 的所有属性的域都是原子的，那么 $\small R$ 属于第一范式 (First Normal Form, 1NF)

    e.g.1 某机构给雇员分配形如 $\small\mathrm{CS012}$ 和 $\small\mathrm{EE127}$ 的雇员号，其中包含了机构和编号两部分信息。从数据库的角度看，如果应用程序没有把标识号拆开并解析系名 (例如单独另设一个属性)，就可以认为这个域是原子的。

    2. 函数依赖

    e.g.2 $\small\mathit{instructor}(\mathit{\textcolor{black}{ID},name,dept\_name,salary})$ $\small\mathit{department}(\mathit{\textcolor{black}{dept\_name},building,budget})$
    假设用 $\small\mathit{inst\_dept}\ (\mathit{\textcolor{black}{ID},name,salary,dept\_name,building,budget})$ 取代上面的两个模式。则会导致两个问题：

    • 同一个系的信息可能被存储多遍，造成冗余
    • 在设计上，不便新建一个系的信息

     为了避免像上面的例子中的冗余的问题，我们定义: “如果存在模式 $\small(\mathit{dept\_name,budget})$，则 $\small\mathit{dept\_name}$ 可作为主码”。这条规则即为函数依赖 $\small\mathit{dept\_name\to budget}$。
     基于此规则，且由于 $\small\mathit{dept\_name}$ 不可能是 $\small\mathit{inst\_dept}$ 的主码 (一对多)，$\small\mathit{budget}$ 也可能会重复，由此判断 $\small\mathit{inst\_dept}$ 不是一个好的设计。
     在下面的描述中，用希腊字母表示属性集，用大写字母表示 (能成为) 模式的属性集。

    (1). 定义

    考虑关系模式 $\small r(R)$，其中 $\small\alpha\subseteq R$，$\small\beta\subseteq R$。

    • 给定 $\small r(R)$ 的一个实例 $\small r'$，若对实例 $\small r'$ 中任意的一对元组 $\small t_1$ 和 $\small t_2$，满足
      $\small\mathbf{if}\ \ t_1[\alpha]=t_2[\alpha]\ \ \mathbf{then}\ \ t_1[\beta]=t_2[\beta]$，则实例 $\small r'$ 满足函数依赖 $\small\alpha\to\beta$
    • 如果 $\small r(R)$ 的每个合法实例都满足函数依赖 $\small\alpha\to\beta$，就说函数依赖在 $\small r(R)$ 上成立 (hold)
    • 若 $\small K\to R$ 在 $\small r(R)$ 上成立，则 $\small K$ 是 $\small r(R)$ 的一个超码
    • 若函数依赖 $\small\alpha\to\beta$ 满足 $\small\beta\subseteq\alpha$，称此函数依赖是平凡的 (trivial)
    需要注意，一个关系模式的实例可能满足某些函数依赖，但它不一定在关系模式上成立。例如某个实例可能满足 $\small\mathit{room\_number}\to\mathit{capacity}$，但在现实世界中，不同的两个教室可能具有相同的房间号，因而不将它加入 $\small\mathit{classroom}$ 的函数依赖集中。

    (2). 运算律

    Armstrong 公理：

    • 自反律 (reflexivity)：若 $\small\beta\subseteq\alpha$，则 $\small\alpha\to\beta$ 成立
    • 增补律 (augmentation)：若 $\small\alpha\to\beta$ 成立，则 $\small\gamma\alpha\to\gamma\beta$ 成立
    • 传递律 (transitivity)：若 $\small\alpha\to\beta$ 和 $\small\beta\to\gamma$ 成立，则 $\small\alpha\to\gamma$ 成立
    上述公理是完备的，给定函数依赖集 $\small F$，它们能产生全部 $\small F^+$，但会很麻烦。
    • 合并律 (union)：若 $\small\alpha\to\beta$ 和 $\small\alpha\to\gamma$ 成立，则 $\small\alpha\to\beta\gamma$ 成立
    • 分解律 (decomposition)：若 $\small\alpha\to\beta\gamma$ 成立，则 $\small\alpha\to\beta$ 和 $\small\alpha\to\gamma$ 成立
    • 伪传递律：若 $\small\alpha\to\beta$ 和 $\small\gamma\beta\to\delta$ 成立，则 $\small\alpha\gamma\to\delta$ 成立

    (3). 属性集的闭包

    若 $\small\alpha\to B$，称属性 $\small B$ 被 $\small\alpha$ 函数确定 (functionally determine)。
    函数依赖集 $\small F$ 下所有被 $\small\alpha$ 函数确定的属性集合称为 $\small F$ 下 $\small\alpha$ 的闭包，记为 $\small\alpha^+$。 

    result = α
do
    for each β → γ in F
        if β ⊆ result
            result = result ∪ γ
while(result 不变)
    右侧算法的时间为 $\small O(|F|^2)$。存在一个 $\small O(|F|)$ 的算法，暂不介绍。
    属性闭包算法可用于：
    • 为判断 $\small\alpha$ 是否为超码，检查是否 $\small R\subseteq\alpha^+$
    • 为判断 $\small\alpha\to\beta$ 是否成立，检查是否 $\small \beta\subseteq\alpha^+$
    • 计算 $\small F^+$: 对任意 $\small\gamma\subseteq R$，求 $\small\gamma^+$；对任意 $\small S\subseteq\gamma^+$，输出一个函数依赖 $\small\gamma\to S$

    3. Boyce-Codd 范式

    (1). 定义

    设 $\small R$ 是具有函数依赖集 $\small F$ 的关系模式，如果对于所有形如 $\alpha\to\beta$ 的函数依赖 ($\small\alpha\subseteq R, \beta\subseteq R$)，下面至少有一项成立：

    • $\small\alpha\to\beta$ is trivial ($\small\beta\subseteq\alpha$)
    • $\small\alpha$ 是 $\small R$ 的一个超码

    则称 $\small R$ 属于 BCNF。
    在例 2 中，$\small\mathit{dept\_name\to budget}$ 在 $\small\mathit{inst\_dept}$ 上成立，但 $\small\mathit{dept\_name}$ 并不是超码。

    (2). 分解

    设 $\small R$ 为不属于 BCNF 的一个模式，则存在至少一个非平凡的函数依赖 $\small\alpha\to\beta$，$\small\alpha$ 不是 $\small R$ 的超码。在设计中用以下两个模式取代 $\small R$:

    • $\alpha\cup\beta$
    • $\small R-(\beta-\alpha)$

    分解之后产生的模式可能有一个或多个不属于 BCNF，这种情况还需要进一步分解，直到得到一个 BCNF 模式集合。

    (3). 判定

    判定一个关系是否属于 BCNF:

    • 检查所有 non-trivial 的函数依赖 $\small\alpha\to\beta$ 是否违反 BCNF: 计算 $\small \alpha^+$，并验证它是否包含 $\small R$ 中的所有属性。
    • 只需检查 $\small F$ 中的函数依赖，可以证明，如果 $\small F$ 中没有函数依赖违反 BCNF，那么 $\small F^+$ 中也不会有。
    但是，分解后只用 $\small F$ 就不够了。

    e.g. 考虑关系模式 $\small R(A,B,C,D,E)$，设 $\small F=\lbrace A\to B,BC\to D\rbrace$。设 $\small R$ 分解成 $\small R_1(A,B)$ 和 $\small R_2(A,C,D,E)$，尽管 $\small F$ 中没有一个只包含来自 $\small (A,C,D,E)$ 的式子，$\small R_2$ 也不满足 BCNF，可通过 $\small F^+$ 中的 $\small AC\to D$ 判断。

    下面这种方法有时会比计算 $\small F^+$ 上的所有函数依赖简单。为检查 $\small R$ 分解后的关系 $\small R_i$ 是否属于 BCNF，应用如下判定：

    • 对 $\small R_i$ 中每个子集 $\small\alpha$，确保 $\small\alpha^+$ 要么不包含 $\small R_i-\alpha$ 的任何属性，要么包含 $\small R_i$ 的所有属性
    如果 $R_i$ 上有某个 $\small\alpha$ 违反此条件，可以证明 $\small\alpha\to(\alpha^+-\alpha)\cap R_i$ 出现在 $\small F^+$ 中，此函数依赖说明 $\small R_i$ 违反 BCNF。
