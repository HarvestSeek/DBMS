    4.intermediate

    1. 视图

     此前的讨论假定给出的关系都实际存储于数据库中，但有时让用户看到整个逻辑层是不合适的。例如，一个职员没有权限看到教师的工资值。
     此外，我们还可能希望创建一个比底层逻辑模型更符合直觉的关系。基于此，SQL 支持通过视图 (view) 定义一个 "虚关系"，它并不预先存储，只在被引用时动态生成。

    (1). 视图定义

     定义视图时，必须给定其名称以及计算视图的查询。其格式为 $\small\mathrm{create\ view}\ v\ \mathrm{as}\ \mathit{[ query}$-$\small\mathit{expr]}$，视图的属性名可以通过 $\small v\ (\mathit{attr_1,...,attr_n})$ 显式指定。
     为了避免视图过期，数据库只存储生成视图的查询表达式，且在它被某个其他查询引用时执行。

    (2). 物化视图

     某些特定的数据库允许视图把结果存储在数据库中，且它们保证：一旦用于定义视图的关系被修改，视图存储的关系也会跟着修改，这样的视图被称作物化视图 (materialized view)。
     频繁使用视图的应用会从物化视图中受益。那些需要快速响应基于大关系上聚集计算的特定查询也会受益，因为它避免读取大的底层关系，使用视图回答查询就更快。

    (3). 视图更新

     视图更新即对视图关系进行更新、插入或删除。更新视图的同时，定义视图的底层关系也需要相应的更新，因此这是一个很麻烦的问题，一般不允许对视图关系修改。
     SQL:1999 对于何时在视图上执行更新有更复杂的规则集，不同的数据库系统也指定了不同的条件以允许更新视图关系。

    2. 事务

     事务 (transaction) 由查询和更新语句的序列组成，SQL 标准规定，当一条 SQL 语句被执行就开始了一个事务。下面两个 SQL 语句会结束一个事务 ($\mathrm{work}$ 关键字可选)：

    • $\small\mathrm{commit\ work}$: 提交当前事务，即永久保存事务对数据库的更新，提交后另一个新的事务自动开始。事务提交之后无法回滚。
    • $\small\mathrm{rollback\ work}$: 回滚事务，即撤销对数据库的更新，数据库恢复到事务第一条语句执行前的状态。

    e.g. 假设 $\small\mathit{student}$ 关系的 $\small\mathit{total\_credit}$ 属性要保持在最新状态，只要学生成功修完一门课程，$\small\mathit{takes}$ 关系就记录学生修完课程的信息，相应的 $\small\mathit{student}$ 也被更新。如果在执行完 $\small\mathit{takes}$ 的更新后崩溃了，那么数据库就会变得不一致，此时就需要回滚。

    在很多 SQL 的实现中，默认方式下每条 SQL 语句自成一个事务，且一执行完就提交。如果一个事务要执行多条语句就必须关闭自动提交，如何关闭取决于特定数据库的实现，诸如 JDBC 等接口存在标准化方式完成此工作，一个比较好的选择是，通过 SQL:1999 标准的 $\small\mathrm{begin\ atomic}\ ...\ \mathrm{end}$ 实现。

    3. 完整性约束

    (1). 单关系约束

    • $\small\mathrm{not\ null}$: 声明在属性定义后，禁止为空
    • $\small\mathrm{primary\ key}(A_1,...,A_n)$: 声明属性构成主码，主码必须非空且唯一
    • $\small\mathrm{unique}(A_1,...,A_n)$: 指出相应属性集构成一个候选码，关系中不存在两个元组在列出的属性上取值相同
    • $\small\mathrm{check}(\mathit{predicate})$: 指定一个谓词，关系中每个元组都必须满足，常用来指定属性值必须满足某个条件。

    (2). 参照完整性

    保证一个关系给定的属性集取值必须在另一关系中出现，如果没有非空约束则可为空。语句格式为

    • $\small\mathrm{foreign\ key}(A_{1},...,A_{1})\ \mathrm{references}\ r$
    • $\small\mathrm{foreign\ key}(A_{11},...,A_{1n})\ \mathrm{references}\ r(A_{21},...,A_{2n})$
    第一种默认被参照属性时被参照表的主码属性，第二种显式指定属性，但要求指定的属性列表必须是被参照表的候选码，即必须用 $\small\mathrm{primary\ key}$ 或 $\small\mathrm{unique}$ 约束。

    当违反参照完整性约束，通常会拒绝破坏完整性的操作。但在 $\small\mathrm{foreign\ key}$ 中可以指明，如果被参照关系由于删除或更新违反了约束，那么系统必须采取一些操作顺应此改变，而不是拒绝。

    e.g.  
    CREATE TABLE course (
    ...
    FOREIGN KEY (dept_name) REFERENCES department
        ON DELETE CASCADE, ON UPDATE CASCADE;
)
    • $\small\mathrm{on\ delete\ cascade}$ : 若删除被参照表的元组导致完整性被违反，级联删除参照表的元组
    • $\small\mathrm{on\ update\ cascade}$ : 若更新被参照表的元组导致完整性被违反，将参照表的元组改为新值
    • $\small\mathrm{set\ null}$ : 完整性被违反则置为空值
    • $\small\mathrm{set\ default}$ : 完整性被违反则置为属性的默认值

    4. 触发器

    触发器 (trigger) 是一条语句，指明了条件与动作。当对数据库的修改满足了其条件，它即被自动执行。其基本格式为：

    $\small\mathrm{create\ \ trigger}\ \ \mathit{tname}\ \ \mathrm{[before \ |\ after]}\ $ <$\small event$> $\small\ \mathrm{on}\ \ \ r$
    $\small\mathrm{for\ \ each\ \ [row\ |\ statement]}$
    $\small\mathrm{when}$ <$\small\mathit{p}$>
    $\small\mathrm{begin} ...\ ...\ \mathrm{end};$
    其中 $\small\mathrm{when}$ 指明触发条件，$\small\mathrm{begin}$ 和 $\small\mathrm{end}$ 之间实现触发器的动作。

    (1). 使用场景

    • 实现无法通过 SQL 约束机制指定的完整性约束
    • 当满足特定条件时，对用户触发警报或执行某项任务
    • 物化视图：修改底层关系时，通过触发器修改视图 (对于那些不支持物化视图的数据库)
    e.g.1 当关系 $\small\mathit{takes}$ 的 $\small\mathit{grade}$ 属性被更新时，意味着一名学生修完了一门课程。此时需要更新 $\small\mathit{student}$ 的 $\small\mathit{total\_credits}$ 属性。 
    CREATE TRIGGER credits_earned AFTER UPDATE OF takes ON (grade)
FOR EACH ROW
WHEN old.grade is null AND new.grade < 100
BEGIN
    UPDATE student
    SET total_credits = total_credits + (
        SELECT credits FROM course 
        WHERE course.course_id = new.course_id
    )
    WHERE student.id = new.id
END;
    e.g.2 当插入 $\small\mathit{student}$ 信息时，如果名字为空串，代表名字不确定，用空值代替。 
    CREATE TRIGGER name_modify BEFORE INSERT ON (student)
FOR EACH ROW
WHEN new.name = ''
BEGIN
    SET new.name = null
END;

    (2). 其他

    • 和插入删除相关的外键约束不应用触发器实现
    • 一个触发器的动作可以除法另一个触发器，避免出现无限的触发器链
